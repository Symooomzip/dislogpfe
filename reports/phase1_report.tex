\begin{sectionbox}[Résumé]
Ce rapport résume les travaux réalisés lors de la \textbf{Phase 1} du projet PFE Dislog. Cette phase couvre la mise en place du projet, le profilage des fichiers CSV sources, la conception et l'implémentation d'un entrepôt de données en schéma en étoile, ainsi qu'un pipeline ETL complet qui extrait, nettoie et charge les données dans SQL Server. Toutes les tables de dimension et de faits sont alimentées et validées. Le pipeline intègre des contrôles de qualité des données (valeurs nulles, doublons, intégrité référentielle) ainsi qu'une capacité de reprise pour les chargements partiels.
\end{sectionbox}

% ============================================================================
\section{Introduction}
% ============================================================================

Le projet vise à construire un système de \textbf{Customer Lifetime Value (CLV)}, de \textbf{Segmentation} et de \textbf{Recommandation} pour Dislog. Les données proviennent de fichiers CSV bruts (ventes, clients, produits, factures, etc.) et doivent être intégrées dans un entrepôt de données dédié pour l'analyse et le reporting.

\begin{infobox}[Périmètre de la Phase 1]
\begin{itemize}[noitemsep,leftmargin=1.2em]
  \item Structure du projet et mise en place de l'environnement
  \item Profilage des données pour comprendre la qualité et la structure des fichiers sources
  \item Conception du schéma en étoile pour l'entrepôt de données
  \item Pipeline ETL : extraction depuis les CSV, transformation (nettoyage, validation), chargement dans SQL Server
  \item Contrôles de qualité et validation des données
\end{itemize}
\end{infobox}

À la fin de la Phase 1, l'entrepôt de données est entièrement alimenté et prêt pour les analyses en aval.

\section{Mise en place du projet}

\subsection{Structure}

Le projet est organisé comme suit :
\begin{itemize}[noitemsep,leftmargin=1.2em]
  \item \texttt{Data/} --- Fichiers CSV bruts (Region, Sector, Customer, Seller, Products, SalesHeader, SalesLine, Invoice). Délimiteur point-virgule ; certains fichiers utilisent l'encodage CP1252.
  \item \texttt{notebooks/} --- Notebooks Jupyter pour le profilage des données.
  \item \texttt{src/} --- Package Python principal. \texttt{src/config.py} contient les chemins, les paramètres CSV et la connexion SQL Server ; \texttt{src/etl/} contient le pipeline ETL complet.
  \item \texttt{src/etl/cleaning/} --- Sous-package pour le staging, la validation, les règles métier, l'intégrité référentielle et les métriques.
  \item \texttt{reports/} --- Rapports de sortie (ce document).
  \item \texttt{StarSchema.sql}, \texttt{Schema.sql} --- Définitions du schéma cible et du schéma source.
\end{itemize}

\subsection{Environnement}

Un environnement virtuel est utilisé avec les dépendances listées dans \texttt{requirements.txt} : pandas, numpy, SQLAlchemy, pyodbc, python-dotenv, matplotlib, seaborn, scikit-learn, xgboost, jupyter, entre autres. La connexion à la base de données est configurée via un fichier \texttt{.env} (ou des variables d'environnement) : \texttt{SQLSERVER\_SERVER}, \texttt{SQLSERVER\_DATABASE}, et optionnellement \texttt{SQLSERVER\_USERNAME} / \texttt{SQLSERVER\_PASSWORD}.

\section{Conception du schéma en étoile}
% ============================================================================

L'entrepôt de données suit un \textbf{schéma en étoile} (star schema) implémenté dans SQL Server et défini dans \texttt{StarSchema.sql}. Cette section explique pourquoi ce modèle a été choisi et détaille chaque table.

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{star_schema.png}
\caption{Diagramme du schéma en étoile --- 6 dimensions autour de 2 tables de faits (FactSales et FactInvoices).}
\label{fig:star-schema}
\end{figure}

\subsection{Pourquoi un schéma en étoile ?}

\subsubsection{Limites du schéma transactionnel source}

Les données brutes de Dislog sont organisées dans un schéma transactionnel classique (OLTP) avec 8 tables reliées entre elles. Ce schéma est \textbf{normalisé} (3NF) : chaque information est stockée une seule fois, sans redondance. C'est adapté aux opérations quotidiennes, mais \textbf{inadapté pour l'analyse} :

\begin{itemize}[noitemsep,leftmargin=1.2em]
  \item \textbf{Trop de jointures} : 5 à 6 tables par requête analytique.
  \item \textbf{Pas de dimension temps} : pas de table calendrier dans le schéma source.
  \item \textbf{Ambiguïté des mesures} : montants dans SalesHeader \textit{et} SalesLine --- risque de double comptage.
\end{itemize}

\subsubsection{Principe du schéma en étoile}

Le star schema est un modèle conçu pour l'analyse (OLAP), issu de la \textbf{méthodologie Kimball} :

\begin{itemize}[noitemsep,leftmargin=1.2em]
  \item \textbf{Au centre} : les tables de \textbf{faits} --- ce qu'on mesure (montants, quantités, paiements).
  \item \textbf{Autour} : les tables de \textbf{dimensions} --- les axes d'analyse (qui, quoi, quand, où, comment).
\end{itemize}

Visuellement, la table de faits centrale est entourée par les tables de dimension, formant une étoile. Chaque requête analytique ne nécessite qu'une jointure entre la table de faits et les dimensions pertinentes.

\subsubsection{Comparaison OLTP vs OLAP}

\begin{center}
\small
\begin{tabularx}{\textwidth}{>{\bfseries}p{4cm} X X}
\toprule
\tableheadcolor
\tableheadfont Critère & \tableheadfont Schéma source (OLTP) & \tableheadfont Star Schema (OLAP) \\
\midrule
Objectif & Opérations quotidiennes & Analyse et reporting \\
\rowcolor{lightbg}

\rowcolor{lightbg}

\bottomrule
\end{tabularx}
\end{center}

\subsection{Concepts clés du design}

\subsubsection{Clés de substitution (Surrogate Keys)}

Chaque table de dimension possède une clé primaire auto-incrémentée (\texttt{CustomerKey}, \texttt{ProductKey}, \texttt{SellerKey}, etc.) au lieu d'utiliser directement les clés naturelles des données source. Ce choix est motivé par :
\begin{itemize}[noitemsep,leftmargin=1.2em]
  \item Les clés naturelles peuvent changer, être longues ou avoir des formats hétérogènes.
  \item Les clés entières permettent des jointures plus rapides.
  \item Elles sont nécessaires pour supporter le SCD Type 2.
\end{itemize}

\subsubsection{Slowly Changing Dimension Type 2 (SCD2)}

Les dimensions DimCustomer, DimSeller et DimProduct intègrent les champs \texttt{ValidFrom}, \texttt{ValidTo} et \texttt{IsCurrent}. Si un client change de région, on conserve les deux versions : l'ancienne (\texttt{IsCurrent=0}) et la nouvelle (\texttt{IsCurrent=1}). Les ventes passées restent associées à la bonne région historique.

\subsubsection{Dimensions dégénérées}

\texttt{SaleID} dans FactSales et \texttt{InvoiceID} dans FactInvoices sont des \textbf{dimensions dégénérées} : des identifiants métier sans attributs propres, stockés directement dans la table de faits pour le regroupement.

\begin{infobox}[Points clés sur les dimensions]
\textbf{DimDate} est \textbf{entièrement générée} : dans le schéma source, les dates ne sont que de simples colonnes. DimDate génère une ligne par jour avec tous les attributs temporels pré-calculés. La clé \texttt{DateKey} au format AAAAMMJJ (ex. 20240315) est un entier pour des jointures rapides.

\textbf{DimCustomer} \textbf{dénormalise 3 tables en 1} : dans le source, il fallait joindre Customer $\to$ Region et Customer $\to$ Sector. Dans le star schema, tout est aplati dans une seule table --- de la redondance volontaire pour éviter des jointures.

\textbf{DimPromotion} et \textbf{DimPaymentMethod} sont \textbf{extraites} de simples colonnes dans SalesLine et Invoice, isolées dans des dimensions dédiées pour normaliser ces attributs.
\end{infobox}

\subsection{Les 2 tables de faits}

\subsubsection{FactSales --- les ventes ($\sim$9,2 millions de lignes)}

Granularité : \textbf{une ligne par ligne de commande} (le niveau le plus fin). Cette table résulte de la jointure SalesHeader $\bowtie$ SalesLine sur \texttt{saleid}.

\textbf{Pourquoi cette granularité ?} On peut toujours agréger vers le haut (par commande, par client, par mois) via des \texttt{GROUP BY}, mais on ne pourrait pas descendre plus bas si les données avaient été stockées au niveau commande.

\textbf{Clés étrangères} (les branches de l'étoile) :
\begin{itemize}[noitemsep,leftmargin=1.2em]
  \item \texttt{OrderDateKey} $\to$ DimDate \hfill\textit{quand la commande a été passée}
  \item \texttt{DeliveryDateKey} $\to$ DimDate \hfill\textit{quand elle a été livrée}
  \item \texttt{CustomerKey} $\to$ DimCustomer \hfill\textit{qui a acheté}
  \item \texttt{SellerKey} $\to$ DimSeller \hfill\textit{qui a vendu}
  \item \texttt{ProductKey} $\to$ DimProduct \hfill\textit{quel produit}
  \item \texttt{PromotionKey} $\to$ DimPromotion \hfill\textit{quelle promotion}
\end{itemize}

\textbf{Mesures} : Quantity, UnitPrice, LineBruteAmount, LineDiscountAmount, LineNetAmount, LineTaxAmount, LineTotalAmount. Toutes au \textbf{niveau ligne uniquement} pour éliminer le double comptage.

\subsubsection{FactInvoices --- les paiements ($\sim$1,4 million de lignes)}

Granularité : \textbf{une ligne par facture}. Table séparée car les factures ont un \textbf{grain différent} des ventes : date de paiement $\neq$ date de commande, méthode de paiement propre, et une commande peut avoir 0, 1 ou plusieurs factures.

\textbf{Clés étrangères} : PaymentDateKey $\to$ DimDate, CustomerKey $\to$ DimCustomer, PaymentMethodKey $\to$ DimPaymentMethod.

\textbf{Mesure} : PaymentAmount.

\section{Pipeline ETL}
% ============================================================================

L'ETL est implémenté en Python sous \texttt{src/etl/}. Flux principal :

\begin{enumerate}[noitemsep,leftmargin=1.2em]
  \item \textbf{\color{accent}Extraction} --- Lecture des CSV bruts via \texttt{load\_csv.py} avec repli d'encodage et normalisation optionnelle de la virgule décimale.
  \item \textbf{\color{accent}Transformation} --- Pour chaque entité, le package \texttt{cleaning/} exécute : normalisation des noms de colonnes, suppression des clés nulles et doublons, application des règles métier (ex. quantité $>$ 0), vérification de l'intégrité référentielle (mapping vers << Unknown >> ou suppression), conversion des types pour SQL Server.
  \item \textbf{\color{accent}Chargement} --- \texttt{star\_loader.py} vide les tables dans l'ordre inverse des FK, puis charge les 6 dimensions et les 2 faits. Insertions par lots (\texttt{chunksize=100}) avec \texttt{fast\_executemany=True}.
\end{enumerate}

\subsection{Notes techniques}

\begin{itemize}[noitemsep,leftmargin=1.2em]
  \item La dimension date est construite dans \texttt{date\_dimension.py} à partir des dates min/max de commande et de livraison de SalesHeader.
  \item FactSales est construit en joignant SalesLine à SalesHeader sur \texttt{saleid}, puis en résolvant toutes les clés de dimension via des lookups et en calculant les mesures au niveau ligne.
  \item FactInvoices utilise un \textbf{lookup basé sur un dictionnaire} de \texttt{salesid} vers (accountid, orderdate) depuis SalesHeader au lieu d'un merge pandas, afin d'éviter l'explosion mémoire.
\end{itemize}

% ============================================================================
\section{Qualité des données \& Validation}
% ============================================================================

Lors du nettoyage, le pipeline :
\begin{itemize}[noitemsep,leftmargin=1.2em]
  \item Supprime les lignes dont les colonnes de clé primaire ou étrangère sont nulles.
  \item Élimine les lignes en doublon (par clé métier ; première occurrence conservée).
  \item Applique les règles métier (ex. SalesLine \texttt{qty} $>$ 0).
  \item Vérifie l'intégrité référentielle : les FK invalides sont mappées vers une ligne << Unknown >> ou supprimées selon la configuration.
\end{itemize}

Les métriques de qualité (row counts après chaque étape, nulls, doublons, violations RI) sont journalisées par le pipeline. La validation finale se fait via \texttt{check\_db} pour confirmer les row counts dans la base.

\section{Problèmes \& Difficultés rencontrés}
% ============================================================================

Bien que la Phase 1 soit terminée et stable, plusieurs problèmes non triviaux ont été rencontrés et ont influencé la conception finale.

\subsection{Limite de paramètres SQL Server}

\begin{warnbox}[Erreur de taille de lot]
Lors du premier chargement des tables de faits avec \texttt{to\_sql}, le pipeline a échoué avec \textit{``The incoming request has too many parameters''}. SQL Server limite le nombre de paramètres par batch ($\sim$2100). Avec 15 colonnes par ligne, le maximum est $\sim$140 lignes/batch.
\end{warnbox}

\textbf{Solution} : Un \texttt{chunksize=100} explicite a été appliqué à tous les appels \texttt{to\_sql}, combiné avec \texttt{fast\_executemany=True} sur le moteur SQLAlchemy.

\subsection{Temps de chargement très long}

Le chargement complet du schéma en étoile --- en particulier \textbf{FactSales} ($\sim$9,2M lignes) et \textbf{FactInvoices} ($\sim$1,4M lignes) --- peut prendre de l'ordre de \textbf{6 heures} sur une machine de développement typique. Le script de reprise (\texttt{run\_star\_etl\_resume.py}) a été introduit pour éviter de relancer ce long chargement lorsque seul FactInvoices doit être rechargé.

\subsection{Erreur mémoire lors du merge de FactInvoices}

\begin{warnbox}[ArrayMemoryError]
La première implémentation utilisait un \texttt{merge} pandas entre Invoice et SalesHeader sur \texttt{salesid}. Les clés manquantes généraient un produit cartésien, épuisant la RAM ($\sim$12 Go).
\end{warnbox}

\textbf{Solution} : Remplacement du merge par un \textbf{lookup dictionnaire} : un mapping compact \texttt{salesid} $\to$ (AccountID, OrderDateKey) construit en mémoire, avec accès O(1) par ligne d'Invoice.

\subsection{Reprise de l'ETL pour FactInvoices}

Un échec après FactSales nécessitait de relancer l'ETL en entier. Un script dédié \texttt{run\_star\_etl\_resume.py} a été ajouté avec des fonctions qui récupèrent les lookups de dimension existants depuis SQL Server, permettant de charger uniquement FactInvoices.

\subsection{Pression mémoire lors du profilage}

Les tables SalesLine (dizaines de millions de lignes) et FactSales dépassent la capacité RAM. Pour le profilage, la solution a été d'utiliser un \textbf{chargement par échantillonnage} (500K lignes) tout en comptant le total via un line count.

\subsection{Encodage et séparateur décimal}

Certains CSV (Sector, SalesLine, Invoice) sont encodés en CP1252 avec la virgule comme séparateur décimal. L'extracteur stocke désormais l'encodage par fichier et normalise les décimales si nécessaire.

\subsection{Clients anonymes dans DimCustomer}

\begin{warnbox}[$\sim$10\,342 clients sans nom]
Un nombre important de clients avaient un \texttt{AccountName} null, vide ou ``Unknown'', rendant les analyses top-clients peu significatives.
\end{warnbox}

\textbf{Solution en deux volets} :
\begin{enumerate}[noitemsep,leftmargin=1.2em]
  \item \textbf{Logique ETL} : dans \texttt{load\_dim\_customer}, \texttt{AccountName} est défini à \texttt{CLIENT\_\{AccountID\}} quand il est null/vide/Unknown.
  \item \textbf{Correction sans recharger les faits} : script \texttt{reload\_dim\_customer.py} qui désactive les FK sur FactSales et FactInvoices, recharge DimCustomer, puis réactive les contraintes. Les tables de faits volumineuses ne sont pas touchées.
\end{enumerate}
